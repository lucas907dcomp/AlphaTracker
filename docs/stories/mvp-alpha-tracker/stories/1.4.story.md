## Technical Notes

- **Critical:** `apostas` RLS policy has BOTH `USING` and `WITH CHECK`. Without `WITH CHECK`, a malicious client could INSERT an aposta pointing to another user's `operacao_id`. The spec draft had this gap — it was fixed by @data-engineer. Verify this in smoke test assertion 5.
- **RESTRICT FK:** `apostas.casa_id → casas.id ON DELETE RESTRICT` — hard deletion of casas with apostas is blocked. The soft-delete pattern (`is_active = false`) is the correct retirement path.
- If `supabase` CLI is not linked to the project, run `supabase link --project-ref {PROJECT_REF}` first. The user must provide their Supabase project ref and service role key.
- If applying migrations manually: order matters — `001 → 002 → 003` (apostas references operacoes and casas).
