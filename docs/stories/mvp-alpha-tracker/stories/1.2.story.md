## Technical Notes

- **Supabase client choice:** Verify whether to use `@supabase/supabase-js` directly or `@supabase/ssr`. For a Vite SPA (no SSR), `@supabase/supabase-js` with `createClient` is correct. `@supabase/ssr` is for Next.js/Remix. See spec Risk-3.
- Column names in types use `snake_case` matching the DB schema — the app maps to `camelCase` where needed in display/form logic, but raw DB types stay `snake_case`.
- `nota on operacoes.data`: the column is named `data` (Portuguese for "date"). In Supabase JS, `row.data` accesses it. This is distinct from the `{ data, error }` destructuring pattern. Consider aliasing in hooks: `SELECT data AS data_operacao`.
- `CenarioPnL.pnl` is a `number` (already resolved from Decimal by the PnL engine before storing in this struct). The PnL engine itself uses `Decimal` internally.
